javascript设计模式之模板方法模式(template method)

发表于 2016-01-13   |   分类于 javascript   |  

模板方法

顾名思义，模板方法就是可以当做模板的方法

好吧，要说清楚模板方法模式就要从继承说起，因为模板方法模式是一个面向继承的设计模式。那么要理解继承，就要从现实生活中出发，来看看模板方法在继承中的应用。

车企造车

与“世界上不会同时存在两个一模一样的ferrari 250GTO”那个时代的造车技术不同，随着科技的迅速发展，各大车企想尽办法节约造车成本，开发了模块化生产平台。这个模块化的平台规定了在该平台上造车的具体流程，经过此流程生产出来的车可以是不同的车型。举个例子，高尔夫7和现款的奥迪A3都是出自大众集团的MQB平台，显然这两个车是不一样的，但他们都用了MQB平台提供的“模板”。在这里，MQB平台就是我们的模板，他规定了在该模板下造一台车所需要执行的步骤，比如说装引擎、装悬挂、装内饰、装附件的顺序。在有些步骤中，车型不同具体的步骤的执行方式就不一样，比如说引擎，就拿同为高尔夫7下面的不同型号来说，都有1.4t、1.8t（北美）、2.0t等等，因此在平台中只是规定了该步骤在整个流水线里面的顺序而已，具体的实现方法需要在具体的车型中来定义。假设高尔夫和A3的悬挂系统是一样的（当然不一样），那么装悬挂这个过程就可以在MQB平台上定义，在具体的车型上就不用重写了。

说了这么多，我们可以来把整个过程装换成一个继承的问题。MQB平台为父类，而高尔夫和A3为子类。父类中有一个模板方法，封装了造车的步骤的执行顺序，叫做init()。在高尔夫和A3类中，重写了装引擎、装内饰、装附件的方法，并且继承了父类的装悬挂、init()方法。子类的实例只用调用init()方法就能按照模板规定的顺序来造出一辆车。下面看例子：

var MQB = function() {};
MQB.prototype.installEngine = function() {};
MQB.prototype.installSuspension = function() {
    // 前独立后非独立悬挂
};
MQB.prototype.installInterior = function() {};
MQB.prototype.installOther = function() {};
MQB.prototype.init = function() {
    this.installEngine();
    this.installSuspension();
    this.installInterior();
    this.installOther();
};

var GolfVII = function() {};
GolfVII.prototype = new MQB();

GolfVII.prototype.installEngine = function() {
    // 1.4t
};
GolfVII.prototype.installInterior = function() {
    // 高尔夫内饰
};
GolfVII.prototype.installOther = function() {
    // R-line包围
};

var A3 = function() {};
A3.prototype = new MQB();

A3.prototype.installEngine = function() {
    // 1.8t
};
A3.prototype.installInterior = function() {
    // A3内饰
};
A3.prototype.installOther = function() {
    // S-line包围
};

var golf = new GolfVII();
golf.init();

var a3 = new A3();
a3.init();

通过上面的例子可以看出来模板方法是针对继承而言的，这里的代码用javascript模拟了传统面向对象语言如java中的类设计模式。在父类MQB中，installEngine()、installInterior()和installOther()方法是空的，因此要在子类中重写之后才能真正地造出一部车来。在java中，是有编译器来帮我们检查有没有在子类中重写父类中的abstract方法的，但是在javascript中，没有重写是不会有任何提示的，问题也不容易被发现，因此为了解决这个问题，我们可以重构父类代码：

var MQB = function() {};
MQB.prototype.installEngine = function() {
    throw new Error("子类没有重写installEngine");
};
MQB.prototype.installSuspension = function() {
    // 前独立后非独立悬挂
};
MQB.prototype.installInterior = function() {
    throw new Error("子类没有重写installInterior");
};
MQB.prototype.installOther = function() {
    throw new Error("子类没有重写installOther");
};
MQB.prototype.init = function() {
    this.installEngine();
    this.installSuspension();
    this.installInterior();
    this.installOther();
};

这样如果子类没有重写父类中的抽象方法的话就会抛出错误，达到跟java差不多的效果，不过要比java晚一点，java在编译时就能知道错误，而javascript中要等到运行的时候才能抛出错误，灵活的代价，没办法。

javascript没有类

说到javascript的继承，大多数时候我们看到的是类似上面例子中的模拟类的设计模式的写法，那样的写法只是用javascript的本质原型链去模拟了一个他本身没有的东西——类。在看似类的外皮之下，实际上还是基于原型链的对象委托，跟传统面向对象中的继承是完全不一样的。在传统面向对象语言中，通过构造函数构造出来的实例都会复制一份父类上的属性和方法的副本，而在javascript中，复制变成了委托，所有的子类实例都是委托的同一个对象，父类的prototype。这里在讨论模板方法模式，就不扯太远了，感兴趣的可以读一读《you don’t know js》，该书作者Kyle Simpson的很多观点都挺有意思的（连Douglas Crockford都敢黑）。
实际上，传统面向对象的语言中之所以要从类的角度来解决问题，很大程度上是因为我们传递的是对象，而对象是从类中实例化来的。但在javascript中，函数也是对象，我们可以直接传递函数而省去定义构造函数、实例化对象等步骤。

javascript中的模板方法模式

下面我们来让上面的例子更javascripty一点：

var MQB = function(obj) {
    var installEngine = obj.installEngine || function() {
        throw new Error("子类没有重写installEngine");
    };

    var installSuspension = function() {
        // 前独立后非独立悬挂
    };

    var installInterior = obj.installInterior || function() {
        throw new Error("子类没有重写installInterior");
    };

    var installOther = obj.installOther || function() {
        throw new Error("子类没有重写installOther");
    };

    return {
        init: function() {
            installEngine();
            installSuspension();
            installInterior();
            installOther();
        }
    };
};

var golf = MQB({
    installEngine: function() {
        // 1.4t
    },
    installInterior: function() {
        // 高尔夫内饰
    },
    installOther: function() {
        // R-line包围
    }
});
golf.init();

var a3 = MQB({
    installEngine: function() {
        // 1.8t
    },
    installInterior: function() {
        // a3内饰
    },
    installOther: function() {
        // S-line包围
    }
});
a3.init();

现在通过MQB()函数而不是MQB“类”来创建了golf和a3两个不同的实例，并且他们都有init()模板方法调用存在于闭包中的方法。现在代码确实更有javascript特色了，但是还有一个问题，就是如果需要造很多很多的golf和a3，不能每次创建实例都把调用MQB()函数传入具体实现方法的那段代码都复制一遍吧，因此我们应该让MQB()函数返回一个复用性更强的函数(构造函数)来创建后续的实例。

var MQB = function(obj) {
    var installEngine = obj.installEngine || function() {
        throw new Error("子类没有重写installEngine");
    };

    var installSuspension = function() {
        // 前独立后非独立悬挂
    };

    var installInterior = obj.installInterior || function() {
        throw new Error("子类没有重写installInterior");
    };

    var installOther = obj.installOther || function() {
        throw new Error("子类没有重写installOther");
    };

    var constructor = function() {};
    constructor.prototype.init = function() {
        installEngine();
        installSuspension();
        installInterior();
        installOther();
    };
    return constructor;
};

var Golf = MQB({
    installEngine: function() {
        // 1.4t
    },
    installInterior: function() {
        // 高尔夫内饰
    },
    installOther: function() {
        // R-line包围
    }
});
var golf1 = new Golf();
golf1.init();
var golf2 = new Golf();
golf2.init();

var A3 = MQB({
    installEngine: function() {
        // 1.8t
    },
    installInterior: function() {
        // a3内饰
    },
    installOther: function() {
        // S-line包围
    }
});
var a31 = new A3();
a31.init();
var a32 = new A3();
a32.init();

实际上，我们在javascript中实现模板方法模式实际上根本没有用到继承，只是返回了一个在闭包中的构造函数而已。传统面向对象语言很多时候需要用到继承很大程度上是因为他们不像javascript可以直接传递函数，因此他们需要类的设计模式来实现很多功能。既然javascript给我们提供了这么多灵活性，我们就需要好好思考什么时候该用什么模式，在javascript中可以怎么实现，而不是一味地照搬其他语言中的模式。
