javascript设计模式之策略模式(strategy)

发表于 2016-01-06   |   分类于 javascript   |  

先从表单验证说起

我认为要理解一个设计模式必须要知道在什么时候适合使用它，因此在介绍策略模式之前（在之后此系列文章中都会采用这个形式），先来说说他能应用的地方之一——表单验证。
表单验证对前端们来说是低头不见抬头见的，因此很有必要用一个设计模式来让它变得更统一，更便于使用、维护、复用。在说设计模式之前，假设我们有如下的表单：

<form action="#" id="registerForm" method="post">
    用户名：<input type="text" name="userName"/ >
    密码：<input type="text" name="password"/ >
    <button>提交</button>
</form>

那么这时候如果我们需要在提交表单到后台之前现在前端验证表单的正确性，从而减轻后台的负担的话，我们就需要编写表单验证代码，简单起见假设我们现在只需要验证用户名是否为非空和最大长度是否超过10，那么我们可能会像下面这样写：

var validate = function(value) {
    if(value === "") {
        return "用户名不能为空";
    }
    if(value.length > 10) {
        return "用户名长度不能超过10";
    }
}
var registerForm = document.getElementById("registerForm");
registerForm.onsubmit = function() {
    var msg = validate(registerForm.userName.value);
    if(msg) {
        alert(msg);
        retuen false;
    }
}

实际上上面的代码我写了一半就不太想继续写下去了，但是为了说明它有多烂我还是保存下来了。

  • 首先，在validate()函数中，用了两个if判断句来实现不同的表单验证策略，这里只有俩看上去没什么，但是实际情况是不可能只有两种验证策略的，给每一个都加上if无疑会使validate()函数变得庞大无比。
  • 在每个if中return的警告用户的消息直接写在了validate()函数里，如果用户想改动，必须要在validate()函数中来改，想要改动最大长度限制也需要在validate()函数里改动，违反开放－封闭原则。这种需要变化的东西也应该让使用者来决定他的值是什么而不应该事先定义好。
  • 验证策略的复用性差，如果想在程序其他地方也用到这样的验证策略，那就只有复制粘贴。

策略模式

策略模式的核心是，定义一系列不同的算法，将它们包装起来，并且可以相互替换。在传统的面向对象语言中，相互替换由于存在类的限制显得要麻烦许多。在javascript中，函数就是对象，因此不必把每个策略定义成对象，把他们定义成函数就能实现互换。策略模式的目的是把算法的实现和算法的使用分离开来，将不同算法放进一个strategy对象中，并且定义一个context对象来实现他们。需要改动或者添加算法的实现的时候，就只用改动或添加strategy对象的方法，而不关context的事，context对象只关心根据用户传入的信息来调用strategy对象中对应的算法。

用策略模式重构表单验证代码

首先定义strategy对象，将所有的算法定义为它的方法：

var strategies = {
    notEmpty: function(value, errMsg) {
        if(value === "") {
            return errMsg;
        }
    },
    maxLength: function(value, length, errMsg) {
        if(value.length > length) {
            return errMsg;
        }
    },
    minLength: function(value, length, errMsg) {
        if(value.length < length) {
            return errMsg;
        }
    }
};

可以看到，这里的每个验证策略比之前的灵活性高了很多，错误提醒消息和长度等需要自定义的参数都能够自定义。由于strategy对象是被context对象调用来实现具体任务的，下面来定义context对象的构造函数：

var Validator = function() {
    this.cache = [];
};

Validator.prototype.add = function(elem, rules) {
    var value = elem.value,
        that = this;
    for(var i = 0; i < rules.length; i++) {
        (function(strategy) {
            var ruleArr = strategy.strategy.split(":"),
                errMsg = strategy.errMsg;
            that.cache.push(function() {
                var rule = ruleArr.shift();
                ruleArr.unshift(value);
                ruleArr.push(errMsg);
                return strategies[rule].apply(elem, ruleArr);
            });
        })(rules[i]);
    }
};

Validator.prototype.start = function() {
    for(var i = 0, l = this.cache.length; i < l; i++) {
        var msg = this.cache[i]();
        if(msg) {
            return msg;
        }
    }
};

在构造函数中，我们能看到Validator的每个实例都有一个cache数组属性来保存需要调用的策略。而具体的保存逻辑在Validator的实例对象的add()方法中实现。我们给add()方法传入两个参数，一个是需要验证的表单元素，一个是具体的验证规则，这里规定验证规则的格式为[{strategy: "maxLength:6", errMsg: "blabla"}, {strategy: "notEmpty", errMsg: "aaaa"}]，同一个元素可以通过在数组中指定多个对象绑定多个验证策略，每个对象的strategy属性指出了策略的名字，如果该策略需要多余的参数，则用:分开。于是通过传入的rules我们可以将对应的strategy推入Validator的实例的cache数组中。通过Validator的实例的start()方法就可以调用所有的验证策略了。
这样我们就将策略的实现和策略的使用分离开来了。strategies对象负责实现，validator对象负责使用。需要添加或者修改策略的时候就只用改动strategies对象。需要绑定和调用验证表单的策略的时候就只用调用validator对象的add()和start()方法。
同时这也是一个高度可复用插件，可以用在各种不同的项目中，只要你需要表单验证，都能根据需要使用它，如果怕全局变量名冲突可以自己加上命名空间。添加自定义验证策略也非常简单，这里简单起见只定义了三种。下面是具体使用方法：

var registerForm = document.getElementById( 'registerForm' );
var validataFunc = function(){
    var validator = new Validator();
    validator.add( registerForm.userName, [{
        strategy: 'notEmpty',
        errMsg: '用户名不能为空'
    }, {
        strategy: 'maxLength:10',
        errMsg: '用户名长度不能大于10位'
    }, {
        strategy: 'minLength:6',
        errMsg: '用户名长度不能小于6位'
    }]);
    var errorMsg = validator.start();
    return errorMsg;
};
registerForm.onsubmit = function(){
    var errorMsg = validataFunc();
    if ( errorMsg ){
        alert ( errorMsg );
        return false;                // 阻止表单提交
    }
};

上面将验证的过程又包装在了validator()函数中，执行后会根据验证结果返回错误信息。这样可以在应用中存在多个表单需要验证的情况下增强插件的复用性。

策略模式的缺点

说了这么多好话也不是说策略模式没有缺点，缺点还是有的。如上面的例子，作为一个插件，暴露给用户的东西越少当然就越容易使用，这叫做最小知识原则，而策略模式看来是违背了这一个原则了。用户在使用的时候必须要知道策略的实现，才能按规定格式正确传入参数，不然插件是不能正常工作的。
