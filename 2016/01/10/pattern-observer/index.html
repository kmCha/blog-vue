javascript设计模式之观察者模式(observer)和中介者模式(mediator)

发表于 2016-01-10   |   分类于 javascript   |  

观察者模式

什么是观察者模式？

说起观察者模式，脑袋里浮现出的概念似乎不是特别清晰，那么换一个直观一点的名字，观察者模式又叫做发布——订阅(publish-subscribe)模式。个人认为这个名字要比观察者模式更直观，可以直接描述两个对象的职能，发布和订阅。观察者模式主要目的是消除对象之间的耦合。相对于在一个对象中显式调用另一个对象，观察者模式可以将这两个对象解耦，消除显式调用给维护和修改程序带来的不必要的开销。

观察者模式的原理

事件冒泡

实际上在我之前的文章“javascript事件管理与自定义事件解耦”中，就运用了观察者模式。在那篇文章中，用编写的通用事件对象来触发冒泡事件实现了发布对象和订阅对象之间的解耦。简单说来就是通过在DOM元素上绑定一个数据对象，将该元素绑定的事件处理程序都引用到数据对象上，然后定义一个调度函数在事件发生时触发事件处理函数。在通用事件对象上有个trigger()方法可以在一个指定的DOM元素上触发任意冒泡事件，因此我们可以在该元素的任意父元素上捕获（订阅）该事件。
上述的方法是通过事件冒泡来削弱发布对象和订阅对象之间的耦合，只要在订阅对象的子元素上发布了一个事件，那么子元素上面所有订阅了该事件的父元素上的事件处理函数都会被调用，跟发布事件的对象显式一个个调用订阅对象的事件处理函数相比，这种用事件回调的方式的实现使得用户并不用关心哪些具体的对象订阅了该事件，只用将事件发布出去，剩下的事件冒泡和事件处理函数的调用会自动实现。

中介对象

事件冒泡在类似DOM对象那种层级分明的情况下很好用，但是当发布对象和订阅对象是通用的对象的时候就不能用了。因此需要编写一种能够应用于任何情况下的观察者模式。在普通的对象中没有了事件冒泡应该怎么来联系发布对象和订阅对象呢，要想解除他们之间的耦合，不难想到可以定义一个处在他们中间的对象，这里把它叫做中介对象。有了中介对象之后，不管有多少发布者，他们不在乎谁订阅了，只用无脑在事件发生的时候将该消息发布在中介那里，而订阅者也不用关心是谁发布了他们的订阅信息，他们只在乎收到的消息的正确性。中间的事情就是中介对象需要做的事了。

模块间的通信

知道了通用的观察者模式的原理——中介对象之后，我们就可以来看一个实际的例子。假如在一个项目中，我负责写一个ajax模块，在该模块执行过后，需要调用来自其他同学的各种模块，比如说插入图片啊、渲染一个组件啊、blablabla。那么我们可（bu）能（yao）这样写：

ajax.success(function(data) {
    appendImg.init(data);
    renderWidget.render(data);
    blablabla.init(data);
});

然后突然有一天需要在ajax成功返回数据之后添加一个新功能，那么写新功能模块的同学就要联系我，让我在我的ajax代码的回调里再添加一条。可想而之这是相当麻烦的。观察者模式就是致力于解决这种问题的，我在编写ajax模块的时候，在成功返回数据之后，只需要发布数据成功返回这样的一条消息，订阅了该消息的组件就会各自执行自己该做的事情。相应订阅的代码也存在于各个组件的内部，不需要让我知道。而需要添加的组件也只需要在内部订阅该消息就能在不用麻烦我的情况下实现跟存在于回调函数之中相同的效果。
下面来看看用观察者模式重构的代码：

var middle = (function() {
    var _events = {},
        _listen,
        _trigger,
        _remove;

    _listen = function(event, fn) {
        if(!_events[event]) {
            _events[event] = [];
        }
        _events[event].push(fn);
    };

    _trigger = function() {
        var event = Array.prototype.shift.call(arguments),
            fns = _events[event];
        for(var i = 0, l = fns.length; i < l; i++) {
            fns[i].apply(this, arguments);
        }
    };

    _remove = function(event, fn) {
        if(!_events[event]) {
            return;
        }
        if(!fn) {
            _events[event] = [];
        }
        else {
            for(var i = 0, l = _events[event].length; i < l; i++) {
                if(fn === _events[event][i]) {
                    _events[event].splice(i, 1);
                    return;
                }
            }
        }
    };

    return {
        listen: _listen,
        trigger: _trigger,
        remove: _remove
    };
})();

var appendImg = (function() {
    middle.listen("ajaxSuccess", function(data) {
        appendImg.init(data);
    });

    return {
        init: function(data) {
            // 初始化代码
        }
    };
})();

var renderWidget = (function() {
    middle.listen("ajaxSuccess", function(data) {
        renderWidget.render(data);
    });

    return {
        render: function(data) {
            // 初始化代码
        }
    };
})();

var blablabla = (function() {
    middle.listen("ajaxSuccess", function(data) {
        balblabla.init(data);
    })

    return {
        init: function(data) {
            // 初始化代码
        }
    };
})();

ajax.success(function(data) {
    middle.trigger("ajaxSuccess", data);
});

可以看到我们定义了一个中介对象middle，最后在我编写的模块中，只用在回调函数中通过中介对象来发布"ajaxSuccess"这个消息，并不用知道谁需要用到。在需要用到这个消息的模块中，编写模块的人需要在其模块内部订阅该消息，当有另外的模块发布该消息的时候，中介对象就会调用模块传进去的回调函数并且传入从发布者那里传过来的数据。

通过观察者模式，我们可以很容易的添加或者删除他们之间的关联。例如如果现在有一个新的模块需要在ajax成功请求之后调用，那么编写那个模块的人就只用在其内部用中介对象的listen()方法订阅"ajaxSuccess"消息，而不用再联系我修改我的代码了。同样，需要删除的时候，也可以通过中介对象的remove()方法来清除一个回调函数或者清除一个消息的所有订阅者，取决于传入该方法的参数中有没有指定特定的订阅者。

下面通过用定时器来模拟ajax请求并且使用我们上面定义的middle中介对象来实现ajax模块和loadImg模块的模块间的通信：

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>模块间通信</title>
</head>
<body>
    <button>点我发送请求</button>
    <p></p>
    <script type="text/javascript" src="../src/module-subscribe.js"></script>
    <script type="text/javascript">
    window.onload = function() {
        var p = document.getElementsByTagName("p")[0];

        var ajax = (function() {
            var _send = function() {
                p.innerHTML = "正在发送ajax请求";
                setTimeout(function() {
                    p.innerHTML = "请求成功返回";
                    middle.trigger("ajaxSuccess", "success");
                }, 3000);
            };
            return {
                send: _send
            };
        })();
        var loadImg = (function() {
            middle.listen("ajaxSuccess", function(data) {
                loadImg.init(data);
            });

            var _init = function(data) {
                alert("加载图片" + data);
            };
            return {
                init: _init
            };
        })();
        document.getElementsByTagName("button")[0].onclick = function() {
            ajax.send();
        };
    };
    </script>
</body>
</html>

中介者模式

什么是中介者模式

中介者模式是通过在对象之间定义一个中介者对象来消除对象间紧密耦合的一种模式。听起来像是在说观察者模式，其实我也是这么觉得的，这俩不仔细想想还真的是不太好区分

一个例子

要想区分这两个模式，先来看个例子。假如我们要写一个订单处理页面，消费者可以选择产品的颜色和数量，在下方会显示消费者所选择的产品颜色和数量，假设我们已经知道产品的库存，那么在产品库存足够的情况下，消费者能够点击“确认订单”来进行购买，否则说明不能点击按钮的原因。因为要在下面显示颜色和数量，那么在选择了颜色和改变了输入的数量之后就需要对显示的元素进行更改，并且进行一些逻辑判断最终确定按钮的状态，所以我们需要在每次选择框元素的onchange事件和输入元素的oninput事件触发之后都需要对DOM元素进行操作。

观察者模式实现

下面先看看用观察者模式来实现这个例子：

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>stock</title>
</head>
<body>
    <select id="color">
        <option value="">选择颜色</option>
        <option value="red">红色</option>
        <option value="black">黑色</option>
    </select>
    <input type="text" id="number">
    <p id="colorInfo"></p>
    <p id="numberInfo"></p>
    <button id="submit" disabled=true>选择颜色输入数量</button>

    <script type="text/javascript" src="stock.js"></script>
    <script type="text/javascript">
    window.onload = function() {
        var stock = {
            "red": 5,
            "black": 8
        };

        document.querySelector("#color").onchange = function() {
            middle.trigger("colorChange", this);
        };

        document.querySelector("#number").oninput = function() {
            middle.trigger("numberChange", this);
        };

        var handleChange = function(obj) {
            var colorObj = document.querySelector("#color"),
                numberObj = document.querySelector("#number"),
                colorInfo = document.querySelector("#colorInfo"),
                numberInfo = document.querySelector("#numberInfo"),
                button = document.querySelector("#submit");

            if(obj === colorObj) {
                colorInfo.innerHTML = "颜色：" + colorObj.value;
            }
            else if(obj === numberObj) {
                numberInfo.innerHTML = "数量：" + numberObj.value;
            }
            if(!colorObj.value) {
                button.disabled = true;
                button.innerHTML = "请选择颜色";
                return;
            }
            if(!numberObj.value || (((numberObj.value - 0) | 0) !== numberObj.value - 0)) {
                button.disabled = true;
                button.innerHTML = "请输入正确的数量";
                return;
            }
            if(numberObj.value > stock[colorObj.value]) {
                button.disabled = true;
                button.innerHTML = "库存不足";
                return;
            }
            button.disabled = false;
            button.innerHTML = "确认订单";
        };

        middle.listen("colorChange", handleChange);
        middle.listen("numberChange", handleChange);
    };
    </script>
</body>
</html>

上面的代码中，我们在颜色选择框的onchange事件的回调函数和数量输入框的oninput事件回调函数中通过观察者middle对象分别对事件colorChange和numberChange进行了发布。想要达到例子的要求，订阅这两个事件的函数是一个庞大的函数，它接收一个对象作为参数，通过对对象参数进行判断来确定是哪一个元素进行了变化，进而进行接下来的操作。其实这个例子既是观察者模式也是中介者模式，它是一个用观察者模式来实现中介者模式的例子。我们可以称这个庞大的handleChange()函数为中介者(mediator)。

中介者模式实现

中介者模式有两种实现，一种是上面的观察者模式实现，另外一种实现是定义一个中介者对象，暴露一些接口给事件的发出者调用，通过传入参数来在中介者对象中进行下一步的操作。下面看看代码：

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>stock</title>
</head>
<body>
    <select id="color">
        <option value="">选择颜色</option>
        <option value="red">红色</option>
        <option value="black">黑色</option>
    </select>
    <input type="text" id="number">
    <p id="colorInfo"></p>
    <p id="numberInfo"></p>
    <button id="submit" disabled=true>选择颜色输入数量</button>

    <script type="text/javascript">
    window.onload = function() {
        var stock = {
            "red": 5,
            "black": 8
        };

        document.querySelector("#color").onchange = function() {
            mediator.change(this);
        };

        document.querySelector("#number").oninput = function() {
            mediator.change(this);
        };

        var mediator = (function() {
            var colorObj = document.querySelector("#color"),
                numberObj = document.querySelector("#number"),
                colorInfo = document.querySelector("#colorInfo"),
                numberInfo = document.querySelector("#numberInfo"),
                button = document.querySelector("#submit");
            return {
                change: function(obj) {
                    if(obj === colorObj) {
                        colorInfo.innerHTML = "颜色：" + colorObj.value;
                    }
                    else if(obj === numberObj) {
                        numberInfo.innerHTML = "数量：" + numberObj.value;
                    }
                    if(!colorObj.value) {
                        button.disabled = true;
                        button.innerHTML = "请选择颜色";
                        return;
                    }
                    if(!numberObj.value || (((numberObj.value - 0) | 0) !== numberObj.value - 0)) {
                        button.disabled = true;
                        button.innerHTML = "请输入正确的数量";
                        return;
                    }
                    if(numberObj.value > stock[colorObj.value]) {
                        button.disabled = true;
                        button.innerHTML = "库存不足";
                        return;
                    }
                    button.disabled = false;
                    button.innerHTML = "确认订单";
                }
            };
        })();
    };
    </script>
</body>
</html>

稍加改动之前用观察者模式来实现的代码就能得到用另一种方法实现的中介者模式。在代码中定义了一个mediator中介者对象，定义了一个change()接口来给外部的对象调用。剩下的实现原理就跟上面的handleChange()函数是一样的了。

中介者模式与观察者模式的异同

通过对观察者模式的总结，可以看出来观察者模式就是在消息的发送方和消息的接收方中间定义了一个观察者对象。在消息的发送者对象中，消息是直接发给观察者对象的。在消息的接收方，消息是通过观察者对象订阅的。因此发送方和接收方不必知道对方，只用知道观察者就行，因此实现了解耦。
而中介者模式如果用观察者模式来实现的话，就可以看成消息的订阅者是一个庞大的中介者对象，而这个对象可以同时订阅很多的消息，通过传入的参数来进行业务的判断。这样消息的发送者不需要知道消息的接收者是谁，通过订阅不同的消息之后消息的接收者（也就是中介者对象）也可以不需要知道消息的发送者是谁，这样也实现了解耦。
如果中介者模式是通过定义中介者对象来实现的话，上面的消息的发送者只需要调用中介者对象暴露出来的接口就行，剩下操作的都在中介者对象中处理。
因此，可以看出观察者模式和中介者模式确实比较接近，但是也存在差别，不然也不会以两个不同的设计模式存在对吧。所以我认为，可以从不同角度来判断什么时候应该使用观察者模式，什么时候应该使用中介者模式。

  1. 观察者模式是站在消息的发送者的角度来看的。通常一个消息可以有很多不同的订阅者，当消息发出之后，不同的订阅者各自处理自己的业务。
  2. 中介者模式是站在消息的接收者的角度来看的。通常一个消息的接收者（中介者对象）可以接收许多不同的消息，并且根据触发的事件来判断需要处理的业务。
