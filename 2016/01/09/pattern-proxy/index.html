javascript设计模式之代理模式(proxy)和装饰者模式(decorator)

发表于 2016-01-09   |   分类于 javascript   |  

代理模式

代理是什么？

代理比较好理解，生活中处处都是代理。例如房屋交易中介，你只管买房，房主只管卖房，中间的一些繁琐的事务由中介来做，这里可以说中介代理了房主来卖房子。
因此，代理在javascript中就可以理解为，要跟一个对象通信，但是由于某种原因需要经过一个中间的代理对象来访问实际想要访问的对象。

为什么要用代理？

我可以访问一个对象为啥偏要在中间加另一个对象？可以保证的是绝不是吃饱了撑着了，要回答这个问题，先来看一个具体的例子——图片预加载。

代理模式实现图片预加载

图片预加载就是加载一个图片的时候，在图片还没有下载完成的那一段空白时期，让一个提前加载好的图片（比如各种loading菊花图）来填充图片元素，当要我们想要加载的图片下载完成后再替换成我们想要的图片。知道了原理那么就来编写代码实现一个图片预加载：

首先是html文件：

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>image-preload</title>
</head>
<body>
    <button>点我加载图片</button>
    <script type="text/javascript" src="../src/image-preload.js"></script>
    <script type="text/javascript">
    window.onload = function() {
        var img = createPreloadImage();
        document.getElementsByTagName("button")[0].onclick = function() {
            img.setSrc("http://7xoxzw.com1.z0.glb.clouddn.com/15-12-13/22567280.jpg");
        };
    };
    </script>
</body>
</html>

这里能看到我们需要编写一个createPreloadIamge()函数然后用它来返回一个img对象，通过调用img对象的srtSrc()方法并传入一个url来设置图片的地址。下面来看看这个怎么实现

普通图片预加载

// image-preload.js
var createPreloadIamge = function() {
    var proxyImg = new Image(),
        loadingImg = new Image(),
        realImg = document.createElement("img");

    document.body.appendChild(realImg);

    loadingImg.src = "http://7xoxzw.com1.z0.glb.clouddn.com/16-1-12/64506391.jpg";

    proxyImg.onload = function() {
        realImg.src(this.src);
    };

    return {
        setSrc: function(src) {
            realImg.src(loadingImg.src);
            proxyImg.src = src;
        }
    };
}

遵循首当其冲原则（一看就是我乱编的好吗），第一个例子必定成为炮灰，那么它的问题在哪？问题在于这段代码不符合单一职责原则（这个是正儿八经的）。什么是单一职责原则？引用曾探老师的话：

单一职责原则指的是，就一个类（通常也包括对象和函数等）而言，应该仅有一个引起它变
化的原因。如果一个对象承担了多项职责，就意味着这个对象将变得巨大，引起它变化的原因可
能会有多个。面向对象设计鼓励将行为分布到细粒度的对象之中，如果一个对象承担的职责过多，
等于把这些职责耦合到了一起，这种耦合会导致脆弱和低内聚的设计。当变化发生时，设计可能
会遭到意外的破坏。

那不符合单一职责原则又怎么了？我还不是能实现图片预加载。是的能实现，但是当createPreloadIamge()函数既担任了创建图片的责任，又担任了图片预加载的责任的时候，那如果以后网速飞起不需要再用预加载了，或者由于某些原因不需要用图片预加载了，需要将这个功能删除，那就麻烦了。要想不改动前面的html文件继续实现图片的创建，必须要从createPreloadIamge()函数中分离出图片创建的那部分，删除掉图片预加载的那一部分，并且还要提供一样的setSrc()接口，想想都觉得麻烦。
下面用代理模式重构上面的代码。

文艺图片预加载

var createImage = function() {
    var img = document.createElement("img");
    document.body.appendChild(img);
    return {
        setSrc: function(src) {
            img.src = src;
        }
    };
};

var createPreloadImage = function(src) {
    var proxyImg = new Image(),            // 用于预加载图片
        loadingImg = new Image(),        // 用于提前加载菊花图
        realImg = createImage();        // 真正的图片元素

    loadingImg.src = "http://7xoxzw.com1.z0.glb.clouddn.com/16-1-12/64506391.jpg";

    proxyImg.onload = function() {
        realImg.setSrc(this.src);
    };

    return {
        setSrc: function(src) {
            realImg.setSrc(loadingImg.src);
            proxyImg.src = src;
        }
    };
};

可以看到我们创建了两个函数，都会返回一个带有setSrc()方法的对象，但实际上要实现预加载用户只会用到createPreloadIamge()函数，这里为了理解方便用了两个全局变量来保存函数，实际可以用命名空间将createImage()函数隐藏起来，只将createPreloadIamge()函数暴露出来。
要实现预加载，我们要调用createPreloadIamge()函数，在createPreloadIamge()函数内部，它只管预加载的部分，创建图片的功能通过调用createImage()函数来实现。这样职责分离之后，在我们不再需要图片预加载的时候，就只用将html文件中的函数名从createPreloadIamge改为createImage，就能只实现创建图片的功能了。
所以，代理对象（函数）要和本体对象（函数）的接口保持一致，这样就能容易地将本体替换为代理，用户也只用放心地请求代理，只用关心是否能得到想要的结果。

代理模式实现缓存

javascript中复杂的计算会是一笔开销，如果能将计算结果缓存起来，下次接收到一样的参数的时候就能直接返回结算结果而不用重新麻烦处理器再算一次。下面用乘法运算来模拟复杂的运算：

var multiply = function() {
    var result = 1,
        args = arguments;
    for(var i = 0, l = args.length; i < l; i++) {
        result *= args[i];
    }
    return result;
};

var proxyMul = (function() {
    var cache = {};
    return function() {
        var argsStr = Array.prototype.join.call(arguments, ",");   // 将传入的参数合为一个字符串作为缓存的键名
        return cache[argsStr] || cache[argsStr] = multiply.apply(this, arguments);
    }
})();

proxyMul(1,2,3); // 6

这样通过乘法函数multiply()的代理函数proxyMul()函数就可以用相同的传入参数实现带有缓存功能的乘法运算了。还可以对proxyMul()函数稍加改动编写一个可以根据传入本体运算函数来生成对应的代理函数的函数：

var multiply = function() {
    var result = 1,
        args = arguments;
    for(var i = 0, l = args.length; i < l; i++) {
        result *= args[i];
    }
    return result;
};

var add = function() {
    var result = 0,
        args = arguments;
    for(var i = 0, l = args.length; i < l; i++) {
        result += args[i];
    }
    return result;
};

var createProxy = function(fn) {
    var cache = {};
    return function() {
        var argsStr = Array.prototype.join.call(arguments, ",");   // 将传入的参数合为一个字符串作为缓存的键名
        return cache[argsStr] || cache[argsStr] = fn.apply(this, arguments);
    }
};

var proxyMul = createProxy(multiply);
var proxyAdd = createProxy(add);

proxyMul(1,2,3); // 6
proxyAdd(1,1,1); // 3

装饰者模式

什么是装饰者模式？

装饰者模式的核心，是动态地给对象加上某些功能，即装饰。而加了功能之后通常会返回一个内部调用原对象或原函数的对象或者函数，附加的功能在返回的新对象中实现，原对象中的功能不受影响。是的，听起来有点像代理模式，同样是通过另外一个对象包装原对象，在原对象的基础功能之上可以附加一些其他的功能。先不急，后面再说他们有啥区别，先来看看装饰者模式。

还是举个例子

例如现在我们有一个之前就写好了的函数doSomething()，但是在现在的项目中的某个地方需要给他动态地添加一个新的功能console.log("blabla")，并且这个功能需要在执行doSomething()之前执行。这个需求可以很容易的就做到，下面看看怎么来写代码。

一般写法

var _doS = doSomething;
doSomething = function() {
    console.log("blabla");
    _doS();
};

对的，一般情况下就这几行代码就足够了，但是这种写法有两个明显的缺陷

  1. 会创建中间变量（这里的_doS）。这里只有一个，当程序中这种需求多了起来，中间变量也会相应增多，维护起来是个问题。
  2. 存在this劫持的问题。上述代码不会出现这种问题，但是在有些地方会有这个问题，看下面例子。

this劫持

如果我们用上面的写法，来给例如document.getElementById()函数增加功能，我们来看看会出现什么问题：

var _getElem = document.getElementById;
document.getElementById = function(id) {
    console.log("blabla");
    _getElem(id);
};

可以看到，document.getElementById引用的函数被引用给了_getElem变量，并且在新的document.getElementById()函数中直接调用了，因此跟之前的作为document对象的方法调用不同，他的this是指向window对象（浏览器中）而不是document对象的，因此上面代码会报错。当然我们也可以通过给中间变量绑定this来解决这个问题：

var _getElem = document.getElementById;
document.getElementById = function(id) {
    console.log("blabla");
    _getElem.call(document, id);
};

但是这样的写法还是会存在上面的第一个中间变量的问题，而且这样写也并不优雅。

装饰者模式写法

下面我们来使用装饰者模式来重构上面的例子，在重构之前，我们先要实现两个通用的函数：

Function.prototype.before = function(fn) {
    var _self = this;
    return function() {
        fn.apply(this, arguments);
        return _self.apply(this, arguments);
    };
};

Function.prototype.after = function(fn) {
    var _self = this;
    return function() {
        var _ret = _self.apply(this, arguments);
        fn.apply(this, arguments);
        return _ret;
    };
};

上面在Function构造函数的原型对象上定义了两个函数，before和after。任何函数都能够调用他们。如果调用before函数，那么需要传入一个新的函数，before函数返回的函数在执行的时候会先执行当时传入它的那个新函数，然后才会执行当时调用before函数的函数，这样就能动态地给一个函数添加新的功能。after跟before一样，不再赘述。
有了这俩函数，我们就可以愉快地重构上面的例子了：

document.getElementById.before(function() {
    console.log("blabla");
});

这样就能很容易地给一个函数动态地添加某些功能，并且不用产生多余的中间变量，也不存在this劫持的问题。这就是装饰者模式的威力。当然，吹毛求疵一点的话，上面的before和after函数是定义在全局构造函数Function上的，可能有些强迫症患者会觉得不舒服，那么把他们抽离出来放在自己的工具函数里也是相当容易的。

KM.before = function(self, fn) {
    return function() {
        fn.apply(this, arguments);
        return self.apply(this, arguments);
    };
};

KM.after = function(self, fn) {
    return function() {
        var _ret = self.apply(this, arguments);
        fn.apply(this, arguments);
        return _ret;
    };
};

通过before和after函数可以对原函数进行装饰，形成一条长长的装饰链。

代理模式和装饰者模式的异同

说了这么多，回到之前的一个问题上，代理模式和装饰者模式看上去挺像的，那么区别到底在哪？
有一个重要的区别就是意图。在代理模式中，代理对象和原对象的引用关系是静态的，静态的意思就是从编写代码的时候就是确定了的。而在装饰者模式中，装饰链和原函数的引用关系是动态的，就是说之前是不确定的，只有在需要的时候我们才用一些手段（比如before和after函数）来动态地形成装饰链。
另一个区别就是，在代理模式中，往往只有代理对象和原对象之间的一层引用，然而在装饰者模式中，装饰链是可以无限延伸的。
