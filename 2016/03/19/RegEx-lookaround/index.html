RegEx正则表达式环视(lookaround)

发表于 2016-03-19   |   分类于 正则表达式   |  

时间为啥过得这么快，距离上篇文章都过了快一个月了，苦也就不诉了，还是忙毕业找工作，忙得已经没觉得忙了。

环视(lookaround)是什么

正则表达式不仅是一个强大的工具，还是一门语言，更是一门艺术。在以前的一篇博文里头记录了一点正则在JavaScript里的应用，那时也刚认识到正则表达式的威力，就写了篇String类的replace()方法妙用：正则表达式+函数，算是试了个水。今天就来记录下才认识的正则表达式的匹配模式之一————环视。

正如上面所说，我理解的环视是正则表达式的匹配模式之一。而如果要给正则表达式匹配模式不负责任地分个类的话，我觉得会是字符匹配和位置匹配。

字符匹配

字符匹配是指会匹配到字符的表达式。就比如说一个最简单的：

var regex1 = /a/;

regex1会匹配字符串中的a字符。

位置匹配

位置匹配不会匹配到任何字符，而是匹配字符串中的某个位置。比如说字符串开头：

var regex2 = /^/;

regex2会匹配字符串第一个字符前面的那个位置，不会匹配任何字符。

环视是位置匹配

环视属于上面两种类型中的位置匹配。也就是说它并不会匹配某个或某些字符，而是匹配位置用的。

环视的种类

同样，环视也有两种类型，顺序环视(lookahead)和逆向环视(lookbehind)。在perl中以上两种类型的环视都能被支持，但是在JavaScript中只支持顺序环视，因此以下就只介绍关于顺序环视的两种分类。

肯定顺序环视(positive lookahead)

肯定顺序环视在正则表达式中，用(?=开头，)结尾，类似非捕获分组用(?:开头和)结尾。中间放的是子表达式。肯定顺序环视的意思是，匹配一个位置，在这个位置之后紧跟着能匹配子表达式的字符串。

否定顺序环视(negative lookahead)

否定顺序环视的书写方法类似，用(?!开头，)结尾。非定顺序环视的意思是，匹配一个位置，这个位置之后紧跟着的字符串不能匹配子表达式。

环视的应用

基本概念说了这么多，看着有点晕，下面来看一个例子————给数字加上千分符。

首先假如我们有一个数字字符串"10000000"，我们需要把它变成"10,000,000"。用我们顺序环视的视角来看，就可以理解为，在每一次匹配中，我们需要找到一个位置，这个位置紧跟着的必须是3的倍数个数字。那么我们就开始写代码：

var num = "10000000";
var regex3 = /(?=(?:\d{3})+)/g;
num.replace(regex3,",");

可以看到我们在replace函数的第二个参数中传入了","。而前面的正则表达式匹配了一个位置，而不是字符，因此这里的replace与其理解为替换，不如理解为插入。运行上面的代码，会得到下面的结果：

',1,0,0,0,0,000'

尼玛说好的千分符呢，这是个啥。别急，来分析一下问题在哪。上面的regex3可以翻译为：匹配一个位置，这个位置的后面紧跟着能匹配3的倍数个数字的字符串，然后在这个位置上插入","。那么程序运行的流程就是：

  1. 看到'10000000'字符串，发现在字符串起始位置后面紧跟着9个数字，是三的倍数，就是他了。于是第一个位置被插入了","，也就是我们看到的在1之前那个,。
  2. 看到',10000000'字符串，从上次匹配完之后的索引开始匹配，相当于看到的是'0000000'字符串，于是发现字符串开头处的位置后面紧跟着6个0（不管最后一个0的事，因为只要能匹配到3的倍数个就算成功），是三的倍数，就是他了！然后又在最前面插入了,。
  3. 以此类推……

然后就成了我们看到的意料之外的结果。想解决这个问题，我们应该弄清楚正则表达式是怎么匹配的。因为正则表达式匹配的时候是从左往右的，而我们期望的是在一个位置上插入,，这个位置到这个数字的末尾中间一共有3的倍数个数字。注意前面高亮的末尾，问题就出在这。要想在从左往右匹配的过程中，一直匹配到末尾，我们因该对顺序环视的子表达式中匹配到数字的末尾。很容易想到可以用$来匹配表达式的末尾。

改进后的代码：

var num = "10000000";
var regex4 = /(?=(?:\d{3})+$)/g;
num.replace(regex3,",");

得到的结果为：

'10,000,000'

ok，搜发搜古得。但是，如果我们需要匹配的字符串的末尾不是数字呢？不要说我为了剧情发展故意找茬，"10000000$"明明就很常见好吗。然后我们将num改成它试试，结果为：

'10000000$'

并没有什么卵用。因为我们上面regex4中匹配了结尾的位置，我们永远找不到一个位置和结尾的位置之间夹了3的倍数个数字，因为总会有个非数字$。那这时候怎么办呢，我们总要想个办法匹配到数字的边界而不是字符串的边界。

这时候否定顺序环视就起作用了，我们可以用否定顺序环视在子表达式中匹配一个位置，在这个位置之后的那个字符不是数字，那么这个位置就是数字的边界。
改进后的代码：

var num = "10000000$";
var regex4 = /(?=(?:\d{3})+(?!\d))/g;
num.replace(regex4,",");

我们来翻译一下regex4的意思：找到一个位置，在这个位置之后有3的倍数个数字字符，并且在3的倍数个数字之后紧跟一个后面不是数字字符的位置。
运行结果为：

'10,000,000$'

恩看上去万无一失了。但是，这个正则表达式还有一个小问题，先自己思考一下。

然后搞个小破坏，在num中多加一个0。代码如下：

var num = "100000000$";
var regex4 = /(?=(?:\d{3})+(?!\d))/g;
num.replace(regex4,",");

然后看结果：

',100,000,000$'

我勒个%……*（@￥！@#，怎么前面又多了个,？因为我们实际上只对匹配的位置后面做了约束，which is 顺序环视的作用，而没有对该匹配位置的前面做约束。如果在perl中，我们可以用(?<=)和(?<!)来进行逆向环视匹配对位置前面做约束。但是遗憾的是JavaScript并不支持逆向环视，因此我们只能想其他办法，而这个办法也很容易想到。因为我们希望加,的前面始终是一个数字，我们在顺序环视匹配的位置前面多匹配一个数字字符就行了，需要注意的是在替换字符串也就是replace函数的第二个参数中需要同时反向引用这个匹配的数字，因为replace函数会将匹配到的文本一同替换掉。

最终版改进后的代码：

var num = "100000000$";
var regex5 = /(\d)(?=(?:\d{3})+(?!\d))/g;
num.replace(regex5,"$1,");

结果：

'100,000,000$'
