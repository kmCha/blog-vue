React-Redux项目单元测试实践总结

发表于 2016-07-18   |   分类于 javascript   |  

在来阿里实习之前就接触过单元测试的概念，并且在自己做的小项目里也做过一些单元测试的尝试，但现在看来也只是些小打小闹，真正能够深入理解单元测试的过程还是在来到阿里接触了大项目之后。

刚来到这里的时候就立马投入到了紧张的项目开发中，在忙了将近一个月之后，项目也终于快要上线了，这时候开发任务少了下来，我和师兄琢磨着给项目集成一套单元测试，以后维护和修改项目代码之后能够自动将所有测试跑一遍，这样哪里出错就能够迅速定位并干掉。

不久前同组的前辈也总结了一套基于React-Redux的单元测试的设计方式:react-redux-router 单元测试方案设计。这篇文章针对测试框架、测试工具以及测试方法做了细致的讲解。因此本文主要内容会集中在跟前辈设计的那套不同的地方，和我在继承和编写单元测试过程中踩到的坑以及总结的需要注意的地方。

工具

跟前辈的工具链中不同的是，我没有用react官方的react-addons-test-utils,而是用了个人认为API更加直观的来自airbnb的react测试库enzyme。该库同样能够配合karma和webpack形成一套单测的工具。至于测试框架、断言库、MOCK和依赖转移库就完全可以按照个人喜好了选择了。另外，在大多数采用webpack打包的项目中都用到ES6来开发的大环境下，单元测试覆盖率工具建议采用基于Istanbul开发的isparta。以下是个人用到的工具：

  • react测试库：enzyme
  • 测试工具：Karma
  • 测试框架、MOCK：jasmine
  • 依赖转移：babel-plugin-rewire
  • 覆盖率：isparta-instrumenter-loader

使用工具过程中的坑

isparta-loader or isparta-instrumenter-loader

我们需要让覆盖率工具读取到的是被webpack编译之前的代码，这样的覆盖率才会有意义，这个在前辈的文章中也有提到，因此Istanbul和isparta本身是不能用的。在社区寻觅一阵之后找到了能够配合webpack使用的isparta-instrumenter-loader，大家也看到了这是个已经deprecated的工具，并且在它下面写着：

This module is unmaintained. Please use deepsweet/isparta-loader

按照文档配置好karma.conf.js之后，出现不能编译ES6的问题，在社区搜索到了相关的正确配置，发现还是出现同样的问题。一番思索无果之后去求助前辈发现前辈使用的正是那个deprecated的isparta-instrumenter-loader。我抱着这个也不会有用的心态试了试发现错误解决了。

所以，deprecated≠不靠谱。

babel-plugin-rewire报错

使用该插件的时候遇到了TypeError: undefined is not a constructor的错误，在issue里找到了解决方法，但问题并没有得到解决。后来发现是因为我把plugins字段写到了.babelrc文件里，应该是写在karma.conf.js的isparta-instrumenter-loader中，这样才能在webpack编译之前读取文件的时候转移依赖，以下是karma.conf.js中相关部分：

webpack: {
  devtool: 'inline-source-map',
  module: {
    preLoaders: [
      {
        test: /\.js$/,
        loader: 'isparta-instrumenter',
        include: [
          path.resolve(__dirname, './src/actions'),
          path.resolve(__dirname, './src/components'),
          path.resolve(__dirname, './src/containers'),
          path.resolve(__dirname, './src/reducers')
        ],
        query: {
          babel: {
            presets: ['es2015', 'stage-0', 'react'],
            plugins: ["rewire"]
          }
        }
      }
    ],
    //...
  },
  //...
}
PhantomJS + enzyme = 山路十八坑

这里的问题始于，karma默认在package.json中将npm test设置为./node_modules/karma/bin/karma start --single-run --browsers PhantomJS，也就说默认测试是跑在PhantomJS中而不是浏览器中的。这样一般情况下是没有问题的，但是一旦测试代码中用到enzyme的render()API就会报错：

TypeError: undefined is not a constructor (evaluating '_cheerio2['default'].load(html)')

这是因为enzyme中的render()部分是包装的cheerio库，而cheerio又跟PhantomJS不兼容的缘故。因此对于这个问题，将PhantomJS改成chrome即可解决，但是改了之后又出现新的错误：

_cheerio2.default.load is not a function

根据在enzyme’s issue中找到的解决方法，又会出现新错误：

target.dispatchEvent is not a function

根据stackoverflow中的解决方法，发现是console.log()和enzyme的render()(也就是cheerio库)联合使用时才会出现。解决方法就是用node的util模块将要log出来的部分包裹一层。比如说：

import {render} from 'enzyme';
import reactComponent from 'xxx';
let container = render(reactComponent);
// true
console.log(require('util').inspect(container, { depth: null }));
// false
console.log(container);

这里的小标题虽然是吐槽PhantomJS和enzyme的，但其实罪魁祸首是暗藏在他们身后的cheerio库，之所以这样写标题是因为发现并且解决问题的过程，是从PhantomJS和enzyme入手的，在这个过程中不断往深挖，最后社区会把我们引向问题的根源。另外，为了简洁的API代价也挺大的，在不断从坑中爬出来的过程中一度有放弃enzyme使用官方测试库的想法，好在没有放弃才能有这样一段填坑的经历分享出来。

写测试

reducer

reducer的测试代码是各个部分最简单的，跟测试普通函数没有太大区别，在前辈的文章中也讲得很清楚了，这里就不废话了。

action

在action这个部分，对于普通的action creator来说，就只是测试一下返回的action对象是不是我们期望的就行，跟测试普通函数也是一个道理。

相比起来，异步action creator更加值得一提。对于测试异步函数，这里先推荐一篇不错的文章：Testing Asynchronous JavaScript。与文章不同的是，我在测试中用jasmine官方的jasmine-ajax库来mock被测代码里的ajax请求，而不是像文章中那样直接修改请求函数（$.ajax()），因为这样就不用再在测试代码里写一些多余的代码了。

通常，我们的异步action creator长这样：

import {fetchData, redirect} from 'xxx';
export function actionCreator(params) {
  return (dispatch) => {
    dispatch({type: 'LOADING'});
    return fetchData('/url', params).then((res) => {
      if (res.code >= 10000 && res.code < 20000) {
        redirect(window.location.origin + "/syserror.htm");
        return;
      } else if (res.code == 0 && res.data) {
        dispatch({
          type: 'SUCCESS',
          data: res.data
        })
      }
    }).catch(rejection => {
      dispatch({
        type: 'ERROR',
        data: {
          msg: rejection.reason || ''
        }
      })
    })
  }
}

那么针对这样的函数该怎么测？我的理解是，覆盖函数的所有可能性，也就是说我们写的测试代码能够让函数的每一行都跑一遍。那么我们怎么才能通过单元测试框架提供给我们的断言库来判断函数的每一行是不是跑过了呢？拿上面的被测代码例子来说，最内层的函数体会返回一个Promise对象，在该对象的then()方法中，有条件语句。我们想要100%覆盖的话，就必须依次进入到两种条件中，也就是根据我们mock的ajax响应来让函数执行不同的条件分支。在每个条件分支中，找到一个被调用的函数，例如例子中的redirect()和dispatch()，然后在测试代码中将他们初始化为spy，监控他们是否被调用或者被调用时传入的参数来确定测试是否通过。

异步action creator测试代码：

import * as Action from "xxx/action";
import {} from 'jasmine-ajax';
describe("actionCreator()", () => {
  let dispatch = jasmine.createSpy('dispatch');
  let redirect = jasmine.createSpy('redirect');
  let creator = actionCreator();
  beforeEach(() => {
    jasmine.Ajax.install();
  });
  afterEach(() => {
    jasmine.Ajax.uninstall();
    dispatch.calls.reset();
    redirect.calls.reset();
  });
  Action.__Rewire__("redirect", redirect);
  it("returns an action whose type is 'ERROR' when the response has no data", (done) => {
    let mockRes = {
      code: 0,
      data: ''
    };
    let promise = creator(dispatch);
    expect(jasmine.Ajax.requests.mostRecent().url).toBe('/url');
    expect(dispatch).not.toHaveBeenCalled();
    jasmine.Ajax.requests.mostRecent().respondWith({
      "status": 200,
      "contentType": 'application/json;charset=UTF-8',
      "responseText": JSON.stringify(mockRes)
    });
    promise.then(() => {
      expect(dispatch).toHaveBeenCalled();
      expect(dispatch.calls.argsFor(0)[0].type).toEqual('ERROR');
      done();
    })
  });

  it("returns an action whose type is 'SUCCESS' when the response code is outside the range of 10000~20000", (done) => {
    let mockRes = {
      code: 0,
      data: {
        content: [
          {
            info: 'dose not matter'
          }
        ],
        context: {
          icon: 'http://www.taobao.com'
        }
      }
    };
    let promise = creator(dispatch);
    expect(jasmine.Ajax.requests.mostRecent().url).toBe('/url');
    expect(dispatch).not.toHaveBeenCalled();
    jasmine.Ajax.requests.mostRecent().respondWith({
      "status": 200,
      "contentType": 'application/json;charset=UTF-8',
      "responseText": JSON.stringify(mockRes)
    });
    promise.then(() => {
      expect(dispatch).toHaveBeenCalled();
      expect(dispatch.calls.argsFor(0)[0].type).toEqual('SUCCESS');
      done();
    })
  });

  it("won't dispatch actions but redirect when server return code inside the range of 10000~20000", (done) => {
    let mockRes = {
      code: 15000
    };
    let promise = creator(dispatch);
    expect(jasmine.Ajax.requests.mostRecent().url).toBe('/url');
    expect(dispatch).not.toHaveBeenCalled();
    expect(redirect).not.toHaveBeenCalled();
    jasmine.Ajax.requests.mostRecent().respondWith({
      "status": 200,
      "contentType": 'application/json;charset=UTF-8',
      "responseText": JSON.stringify(mockRes)
    });
    promise.then(() => {
      expect(redirect).toHaveBeenCalled();
      expect(dispatch).not.toHaveBeenCalled();
      done();
    })
  })
})

这样这个异步action creator就会有100%的覆盖率了。

component

在测试“高阶”(与state有connection的)component这里，我用的方法和前辈不同。同样这里先推荐一篇文章：Hello Redux 3/3 Unit Test。我测试“高阶”component的方式就是文中指出的：

在測試 smart component 的時候，根據 Redux document 上面的建議是，繞過 connect 的行為，直接測 component 的部份。

剩下的测试方法就跟我在上面写到的测试异步action的方法大同小异了，无非是找到关键函数并且创建spy监控调用。在此之上，我们还需要将该组件中引入的其他“低阶”组件rewire到我们mock的react组件上。

需要注意的是，组件里的state以及props跟组件的视图是存在一个双向关系的。

  • 一方面，我们需要在测试中根据情况通过setState或者setProps来设置组件state进而控制条件渲染分支能够被执行到，具体体现在在enzyme包装之后的实例中找不找得到相关的virtual DOM实例或DOM实例或HTML片段（根据调用的API不同而不同）。
  • 另一方面，我们需要模拟视图中绑定的事件来触发相应的事件处理函数，如果事件处理函数中对state有做修改（大部分情况都有，这里有个疑问，留在后面说），那么我们就需要在断言中判断state来决定测试用例的pass与否。

被测component：

import React from "react";
import {connect} from "react-redux";
import cx from "classnames";
import Layout from "xxx";
import * as Action from "xxx";
import "./index.less";

class Component extends React.Component {
  static defaultProps = {
    user: {},
    common: {}
  }

  constructor(props, context) {
    super(props, context);
    this.state = {toggled: false};
  }

  componentDidMount() {
    dispatch(Action.getInitialData());
  }

  componentWillReceiveProps(nextProps) {
    this.setState({
      user: nextProps.user,
      common: nextProps.common
    });
    this.forceUpdate();
  }
  _handleClick() {
    this.setState({
      toggled: !this.state.toggled
    })
  }
  render() {
    let {toggled} = this.state;
    let customClass = cx('left-nav-section', {toggled: toggled});
    return (
      <Layout {...this.props}>
        <div className="toggle" onClick={this._handleClick.bind(this)}></div>
        <div className={customClass}></div>
      </Layout>
    )
  }
}
export { Component };
export default connect((state)=> {
  return {
    user: state.component.user,
    common: state.component.common
  }
})(component);

测试代码：

import React from 'react';
import { shallow, mount, render } from 'enzyme';
import Container, { Component } from 'xxx/component';

describe("<Component />", function() {
  let props, Layout;
  beforeEach(() => {
    props = {
      dispatch: jasmine.createSpy('dispatch')
    };
    Layout = React.createClass({
      render () {
        return (<div>mock Layout</div>)
      }
    });
    Container.__Rewire__('Layout', Layout);
  });

  it("contains <Layout />", () => {
    expect(shallow(<Component />).find('Layout').length).toEqual(1);
  });

  it("invokes componentDidMount() after mounted", () => {
    spyOn(Component.prototype, 'componentDidMount').and.callThrough();
    let container = mount(<Component {...props}/>);
    expect(Component.prototype.componentDidMount).toHaveBeenCalled();
  });

  it("invokes componentWillReceiveProps() after props changed", () => {
    spyOn(Component.prototype, 'componentWillReceiveProps').and.callThrough();
    let container = mount(<Component {...props}/>);
    container.setProps({
      ...props,
      dispatch: () => 'aa'
    });
    expect(Component.prototype.componentWillReceiveProps).toHaveBeenCalled();
  });

  it("will toggle the state toggled if .toggle element is clicked", () => {
    let container = shallow(<Component {...props}/>);
    expect(container.state('toggled')).toBe(false);
    let element = container.find(".toggle").at(0);
    element.simulate('click');
    expect(container.state('toggled')).toBe(true);
  });

  it("renders .left-nav-section element with or without class 'toggled' according to the state toggled", () => {
    let container = shallow(<Component />);
    expect(container.find('.left-nav-section').length).toBe(1);
    expect(container.find('.toggled').length).toBe(0);
    container.setState({
      toggled: true
    });
    expect(container.find('.left-nav-section').length).toBe(1);
    expect(container.find('.toggled').length).toBe(1);
  });
});
测试component的疑问

经过以上研究和总结，在测试component的部分还存在两点疑问没能解开，希望遇到的前辈大神们能够不吝赐教，在这里先谢谢了:)

第一个疑问：也是我在前文中提到过的疑问，就是在组件内部,除了constructor和componentWillReceiveProps等跟生命周期相关的方法之外，还有许多我们自定义的用于绑定到元素时间或者传给子组件的方法，这些方法是不能通过在组件的prototype上创建spy来监控是否被调用了，因为就算被调用了在断言里也会被当做没有调用的来处理。对于这种情况，如果该方法中修改了组件的state还好处理，在测试代码中读取修改之后的state就能判断该方法是否被调用。如果方法中没有改变state，也可以处理，那就是将该方法中的表达式提取出来，作为依赖引入该组件，再在测试代码中rewire这个依赖，就能够创建spy监控调用情况。我的疑问就是，除了这样麻烦的方法之外，还有什么方法呢？或者是我的思路本身就不对？

第二个疑问：对于引用的外部组件库，比如集团内部的react-sui库。引用Dropdown组件之后，会给组件传入一个特性，值为Menu组件，在Menu组件内部又是若干Menu.Item组件，每一个都绑定了onclick事件。问题就在，Dropdown组件是惰性加载的，只有当鼠标hover到组件身上才会第一次渲染内部的Menu。因此如果在测试代码中什么都不做的话，Menu中的子元素是根本获取不到的，从而也就不能使测试代码覆盖到绑定的事件处理函数上。要想获取到内部的子组件，还必须考虑一个事情，就是测试库针对react的浅渲染(enzyme中是shallow()函数)是不能够触发惰性加载的，因为子组件要真正插入到页面中，必须是以DOM的形式存在的，因此要想获得子组件就必须要用“深”渲染(enzyme中是mount()函数)来将react组件渲染为DOM之后才能获取。那么问题就来了，我们引入的外部组件在.js(x)文件中是以react组件的形式而不是DOM的形式写入的，我们要获取实际渲染出来的DOM的形式就必须翻阅源码或者在浏览器中用devtool来定位到对应的DOM，而且就算找到了对应的DOM，也不能从DOM中读取到react组件上绑定的事件处理函数，那么测试代码就没法覆盖那一部分代码了。我的疑问是有没有什么好办法能在这种情况下获取react组件上绑定的事件处理函数。

其他部分的测试

针对react-redux项目的其他部分的单元测试方法在前辈的文章中已经说得很清楚了，我也没有其他好的心得要补充，这里就不废话了。

集成到IDE(webstorm)

单元测试能在terminal中通过一行命令跑起来已经很方便了，能不能让整个过程变得更方便？答案是肯定的，webstorm官方就有教程能够Karma到项目中，只需要按一下鼠标就能跑整个测试，并且能在IDE中看到每个文件的代码覆盖率，在文件中每行都能提示是否在上一次测试中被覆盖，从而对症下药，非常直观。

总结

通过这段时间对单元测试的编写和研究，总结出了以下心得：

  • 编写单元测试，对一个项目的后期维护很重要，写测试代码的成本跟没有单元测试的项目的后期维护成本相比是很小的
  • 虽然不是正宗的崇尚“测试先行”的TDD，但是在开发完成之后再来编写单元测试除了能够带来上述好处之外，还能在写测试的时候review代码，过程中一方面能够或多或少发现一些测试同学测不出来的bug，另一方面还能发现代码中可以抽取出来做优化的部分。

可见，如今单元测试能在前端遍地开花绝非偶然，存在即合理，花点时间就能essentially缓解后期维护代码的痛苦，何乐而不为？
